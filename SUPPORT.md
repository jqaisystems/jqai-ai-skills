## Public Issues

Open a public GitHub issue when the topic is safe to share publicly.

Good public issues include:

- a skill request using fictional or generalized examples;
- an install problem with private values removed from logs;
- a docs problem, broken link, typo, or unclear instruction;
- a validation issue that does not include private files or secrets.

Do not post credentials, tokens, account details, client data, raw prompts, logs, exports, screenshots with private UI, or local machine paths in public issues.

## Sensitive Reports

Use [`SECURITY.md`](SECURITY.md) for sensitive reporting guidance.

If a report includes a possible credential, private project detail, private source, account identifier, or client-identifying material, do not put it in a public issue.

## What To Include

For install or bug reports, include:

- your operating system;
- the tool you are installing into;
- the install method you used;
- the skill name;
- what you expected;
- what happened instead;
- any relevant logs after removing private values.

For skill requests, include:

- the workflow you want to repeat;
- the type of public-safe input;
- the desired output;
- where human review should happen;
- any safety boundaries the skill must respect.
