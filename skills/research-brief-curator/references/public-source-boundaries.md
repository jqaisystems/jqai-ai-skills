# Public Source Boundaries

Use this reference before turning links, notes, newsletters, or research snippets into a brief.

## Safe Inputs

- Public webpages, public docs, public changelogs, public release notes, and public blog posts.
- User-written notes that do not include private names, account data, credentials, raw prompts, exports, logs, or client material.
- Fake sample records made for examples or demos.
- Approved excerpts that stay short and are linked or cited.

## Unsafe Inputs

- Private dashboards, browser tabs behind login, support tickets, invoices, analytics exports, CRM data, email inboxes, chat logs, internal docs, or client workspaces.
- API keys, tokens, cookies, session values, `.env` files, raw prompts, memory files, local paths, database rows, spreadsheets, CSV exports, or scraped private data.
- Screenshots showing account menus, emails, admin URLs, customer data, private project names, filenames, or local machine paths.
- Full article copies, long source excerpts, or paywalled/private text intended for public reuse.

## Visibility Labels

- `public_ready`: safe to summarize after human review. Must have a public source link or citation.
- `needs_review`: useful but needs fact-checking, source verification, tone cleanup, or quote reduction.
- `private_research`: useful internally, but not appropriate for public sharing.
- `archive`: duplicate, stale, background, or not relevant enough for the current brief.

## Decision Rule

If an item could reveal a private account, client, source list, system configuration, prompt, file path, dataset, lead list, or business strategy, label it `private_research` or remove it from the public brief.
