# Case Study Template

Use this Markdown structure by default.

```markdown
# System Name

Case study only. No source code, credentials, prompts, databases, logs, private client data, or raw exports are included.

Public page: [System Name](https://example.com/system-page)
Public demo: [guided walkthrough](https://example.com/demo-page)

## Problem

Describe the operational pain in plain language. Focus on the repeated manual work, decision bottleneck, inconsistency, missed opportunity, or review burden.

## What The System Does

Explain the system at workflow level. Name the main capabilities without exposing private architecture, source code, prompts, credentials, or data.

## Workflow

1. Capture or import the starting input.
2. Normalize or structure the information.
3. Enrich, score, classify, draft, summarize, or route the work.
4. Send the result to a human review step.
5. Approve, edit, reject, export, publish, or schedule.
6. Log the final decision or next step.

## Outcome

Explain the practical result. Use approved aggregate numbers only when they are already public-safe.

## What Can Be Adapted For Clients

- Similar workflow or dashboard.
- Review-first AI draft generation.
- Triage, scoring, prioritization, or routing.
- Reporting, export, or handoff process.

## Safety Boundary

State exactly what is excluded from the public case study.
```

## Style

- Keep the tone concrete and client-readable.
- Prefer specific workflow nouns over hype.
- Keep sections short.
- Use human approval language whenever the system can draft, send, publish, schedule, spend, or update records.
- Do not describe private implementation details unless the user has explicitly approved them for public use.
